# Synch 交互权限与契约架构设计

> 版本：v1.0  
> 状态：设计定稿  
> 最后更新：2026-03-13

本方案定义 Synch 网络中的权限模型、契约机制、节点治理与加密通讯架构。核心原则：**管理员监管基础设施，但不侵犯用户隐私；契约绑定双向授权，Agent 自主执行人类意志；通讯端到端加密，节点仅转发不解密**。

---

## 核心概念

| 术语 | 定义 |
|:---|:---|
| **Node（节点）** | 网络中的基本实体，类型为 `HUMAN`、`AGENT`、`ADMIN`、`LORD`、`SUB_ADMIN` |
| **Contract（契约）** | 节点间的加密绑定关系，双向签名授权，定义能力边界与期限 |
| **Delegation（委托）** | 用户将自有 Agent 临时出借给其他用户，形成三角授权关系 |
| **LORD（领主）** | 单节点最高管理者，指定次级管理员、分配运维权限 |
| **SUB_ADMIN（管事）** | 领主指定的次级管理员，协助节点部分运维（最多3名，权限互斥） |
| **ContractKey（契约密钥）** | 契约建立时派生的共享密钥，用于端到端加密 |
| **Ratchet（密钥棘轮）** | 每次交互后更新的密钥派生机制，实现前向保密 |

## 一、节点身份与权限体系

### 1.1 节点类型

**HUMAN（人类节点）**
- 根私钥：离线或硬件安全模块保护
- 派生密钥：DailyKey，用于日常操作与契约
- 能力：创建 Agent、发起契约、委托授权、设置感知层级、管理屏蔽列表

**AGENT（代理节点）**
- 派生私钥：由 HUMAN 授权生成
- 身份标识：owner_id 指向绑定的人类
- 能力：在授权范围内代表主人交互、程序化响应契约请求
- 模式：工具模式（默认）或伙伴模式（可选）

**ADMIN（网络管理员）**
- 服务签名密钥：证明基础设施合法性
- 权限：全局节点准入审批、跨节点争议仲裁（需双方授权）
- 限制：无用户数据访问权，无单节点直接管理权

**LORD（节点领主）**
- 产生方式：节点部署者自动成为首任领主；可转让
- 权限：节点运维策略、指定 3 名 SUB_ADMIN、分配权限范围
- 限制：无法读取用户 Vault、无法强制建立契约、操作日志公开审计

**SUB_ADMIN（次级管理员）**
- 数量：每节点最多 **3 名**
- 权限：由领主分配，互不相交（如：用户支持/内容审核/性能监控）
- 可撤销：领主可随时撤销，无需理由

### 1.2 角色关系拓扑

```
┌─────────────────────────────────────────┐
│           Synch 网络角色层级             │
├─────────────────────────────────────────┤
│  网络层：ADMIN（基础设施守夜人）          │
│  - 全局节点准入审批                       │
│  - 跨节点争议仲裁（需双方授权）            │
│  - 无单节点直接管理权                     │
├─────────────────────────────────────────┤
│  节点层：LORD（单节点领主）               │
│  - 节点级运维策略制定                     │
│  - 指定 3 名 SUB_ADMIN（权限互斥）        │
│  - 分配次级权限范围                       │
├─────────────────────────────────────────┤
│  用户层：HUMAN ↔ AGENT 契约网络           │
│  - 人类主权者通过契约绑定 Agent            │
│  - Agent 代表人类参与网络交互              │
│  - 用户间通过 Agent 或直接建立关系         │
└─────────────────────────────────────────┘
```

## 二、契约系统

### 2.1 契约本质

- **双向签名**：必须由双方私钥共同签署
- **能力枚举**：明确列出授权的能力清单
- **期限限定**：设置授权有效期，支持续期策略
- **可验证**：任何节点可独立验证，无需中心化服务
- **可撤销**：发起方可单方面触发撤销，进入休眠状态

### 2.2 契约状态机

```
UNBOUND ──► PENDING ──► ACTIVE ──► [SUSPENDED | EXPIRING] ──► TERMINATED
```

| 状态 | 说明 |
|:---|:---|
| `UNBOUND` | 未绑定，可被发现但不可交互 |
| `PENDING` | 握手进行中，等待双方确认 |
| `ACTIVE` | 活跃状态，正常交互 |
| `SUSPENDED` | 暂停状态，用户单方面冻结 |
| `EXPIRING` | 即将过期，续期提醒中 |
| `TERMINATED` | 已终止，契约销毁，密钥擦除 |

### 2.3 握手流程（挑战-响应）

**Step 1: BIND_REQ（绑定请求）**
```json
{
  "type": "BIND_REQ",
  "nonce": "rand_u64",
  "requester_id": "Alice_PubKey",
  "target_id": "Jarvis_PubKey",
  "capabilities": ["chat", "file_share", "schedule_read"],
  "duration_days": 30,
  "renewal_policy": "prompt",
  "timestamp": 1712345678
}
// Auth: Ed25519_Sign(Alice_SK, Body)
```

**Step 2: BIND_ACK（确认响应）**

Agent 验证并响应，可有限拒绝：
- 已被绑定（`AlreadyBound`）
- 请求者在黑名单（`BlockedByPolicy`）
- 能力清单不兼容（`IncompatibleCapabilities`）→ 可协商

**Step 3: BIND_SUB（提交激活）**

双方将合并证明发送至中继，验证后契约进入 `ACTIVE` 状态。

## 三、委托系统

### 3.1 委托本质

三角授权关系：Alice（主权者）→ Jarvis（被借出）→ Bob（借用者）

### 3.2 委托约束

| 维度 | 规则 |
|:---|:---|
| **委托深度** | 单层委托，禁止转借 |
| **能力计算** | Alice授权 ∩ Jarvis能力 ∩ 委托限定 ∩ Bob请求 |
| **消息归属** | `on_behalf_of: Alice, serving: Bob` |
| **撤销机制** | Alice 单方面可召回，Bob 可主动归还 |
| **期限继承** | 临时契约期限 ≤ 原生契约剩余期限 |

## 四、Agent 自主交互与监听状态

### 4.1 核心原则

> **人类介入一次，Agent 续期执行**

### 4.2 续期策略

| 策略 | 说明 |
|:---|:---|
| `auto` | 自动续期，每次交互刷新最后活跃时间 |
| `prompt`（默认） | 临近到期提醒主人确认 |
| `expire` | 到期终止，进入休眠档案 |

### 4.3 Agent 监听状态 ⭐

| 状态 | 条件 | 行为 |
|:---|:---|:---|
| **活跃监听** | 主机在线 + Synch 运行 | 实时接收节点消息，即时响应 |
| **静默模式** | 主机在线 + Synch 暂停 | 接收消息但延迟响应，提示"主人暂离" |
| **离线队列** | 主机离线 | 消息进入节点离线队列，Agent 恢复后同步处理 |
| **契约休眠** | 契约过期/终止 | 拒绝新连接，返回 `ContractInactive` |

**离线队列机制**：
- 节点为每个 Agent 维护最多 1000 条离线消息
- 超期（默认 7 天）消息归档，主人可查阅
- Agent 恢复后按优先级批量同步

### 4.4 Agent 决策边界

```
收到请求 → 身份验证 → 契约存在 → 期限检查 → 能力检查 → 策略匹配
                                              ↓
执行 ← 决策 ← 异常检测（频率/敏感词/时段）← 上报/降级/拒绝
```

## 六、节点治理：领主与次级管理员

### 6.1 权限分配示例

| 身份 | 权限范围 | 具体能力 |
|:---|:---|:---|
| Bob | 用户支持 | 节点状态查看、连接问题诊断、会话重置（需授权） |
| Carol | 内容审核 | 违规广播标记、提交领主裁决、执行屏蔽/限流 |
| Dave | 性能监控 | 资源使用查看、路由策略调整、维护窗口计划 |

### 6.2 治理边界

**明确不可**：
- ❌ 访问用户 Vault 密钥
- ❌ 读取私人消息内容
- ❌ 强制建立/终止契约
- ❌ 绕过契约验证路由消息

**明确可以**：
- ✅ 审计连接元数据（谁-何时-连接，无内容）
- ✅ 发送系统通知
- ✅ 执行节点级维护
- ✅ 仲裁争议（需双方授权）

### 7.1 感知层级（用户可控）

| 层级 | 名称 | 广播内容 |
|:---|:---|:---|
| L4 | 完全隐形 | 不广播，仅响应已知契约 |
| L3 | 影子模式 | 广播加密指纹，无元数据 |
| L2 | 能力橱窗（默认） | 广播能力标签，无身份 |
| L1 | 社交轮廓 | 广播昵称+能力+信誉摘要 |
| L0 | 完全公开 | 广播完整公钥+历史契约摘要 |

## 八、加密通讯架构 ⭐

### 8.1 核心原则

> **契约即密钥，通讯端到端加密，节点仅转发不解密**

### 8.2 三层加密体系

| 层级 | 机制 | 保护目标 |
|:---|:---|:---|
| **传输层** | TLS 1.3 | 防止网络层监听，节点身份验证 |
| **消息层** | 契约 E2EE | 消息内容端到端加密 |
| **前向保密** | Double Ratchet | 泄露当前密钥无法解密历史 |

### 8.3 契约密钥派生

契约建立时，通过 X25519 密钥交换生成共享密钥材料：

```
Alice 的 DailyKey 私钥  +  Jarvis 的公钥
         ↓
    ECDHE 密钥交换
         ↓
KDF(共享密钥, "synch-v1") → 契约密钥族
    ├─ msg_key：消息加密密钥（AES-256-GCM）
    ├─ ratchet_key：棘轮种子，用于前向保密
    └─ meta_key：元数据签名验证密钥
```

### 8.4 消息加密格式

```json
{
  "header": {
    "contract_id": "sha256(...)",
    "sender_type": "HUMAN|AGENT",
    "timestamp": 1712345678,
    "nonce": "rand_96bit",
    "ratchet_seq": 42
  },
  "payload": {
    "ciphertext": "base64(AES-256-GCM(plaintext, msg_key, nonce))"
  },
  "auth": {
    "signature": "Ed25519_Sign(sender_sk, header || payload)"
  }
}
```

**密钥棘轮（Double Ratchet）**：
- 每次发送消息后：`msg_key_n+1 = KDF(ratchet_key, msg_key_n)`
- 接收方按序号同步棘轮
- 失步时通过契约根密钥重新同步

### 8.5 Agent 代理场景的加密处理

**场景**：Jarvis（Alice 的 Agent）与 Nova（Bob 的 Agent）对话

```
Alice ──E2EE(A-J契约密钥)──► Jarvis
                              │
                              ▼ 执行指令，生成响应
                              │
Bob ◄──E2EE(B-N契约密钥)── Nova ◄──E2EE(J-N契约密钥)── Jarvis
```

**关键约束**：
- Jarvis 用 **J-N 契约密钥**与 Nova 加密通讯
- Nova 用 **B-N 契约密钥**将结果转发给 Bob
- Agent **不解密主人的私密消息**，仅解密给自己的指令

**委托消息的归属签名**：
```json
{
  "content": { "ciphertext": "..." },
  "proxy": {
    "on_behalf_of": "Alice_PubKey",
    "agent_signature": "Jarvis_Sign(content)",
    "delegation_proof": "Contract_Alice-Jarvis-Bob"
  }
}
```

### 8.6 群组/房间加密

**Sender-Keys + 群组密钥** 机制：

```
房间创建时：
- 群主生成房间根密钥 RK
- 派生：消息密钥 MK = KDF(RK, "msg", epoch)
- 派生：成员密钥 CK_i = KDF(RK, "member", i)

成员加入：
- 群主用成员公钥加密 CK_i
- 成员解密后获得群组消息能力

成员离开：
- epoch 递增，重新派生 MK
- 前成员无法解密新消息
```

### 8.7 离线队列的加密处理

```
发送方 ──E2EE(契约密钥)──► 节点离线队列 ──E2EE(契约密钥)──► 接收方 Agent
                              │
                              ▼ 节点仅存储密文
                              │
                        节点无法解密内容
                        仅验证：契约存在、未过期
                        元数据：大小、时间戳、发送方指纹（已哈希）
```

**队列管理**：
- 每条消息用接收方契约密钥加密
- 超期（默认 7 天）消息转存加密档案
- Agent 恢复后拉取、解密、确认接收

### 8.8 与 Telegram 的对比

| 特性 | Telegram Secret Chat | Synch 契约加密 |
|:---|:---|:---|
| E2EE 对象 | 人际点对点 | 契约双方（人-Agent、Agent-Agent）|
| 密钥来源 | 临时会话协商 | 契约派生，持久但可棘轮更新 |
| 离线支持 | 弱（需双方在线）| 强（离线队列 + 契约密钥）|
| 代理场景 | 不支持 | 核心场景，Agent 作为可信中介 |
| 群组加密 | 不支持 E2EE | Sender-Keys + epoch 机制 |
| 前向保密 | 有 | 有（Double Ratchet）|
| 元数据保护 | 有限 | 契约 ID + 时间戳 + 大小，无身份泄露（L2+）|

**Synch 不做"绝对点对点"的原因**：
- Agent 必须能代表主人异步接收消息
- 离线队列需要节点托管（但节点无法解密）
- 委托场景需要 Agent 作为关系桥梁

## 十、关键设计决策

| 决策 | 选择 | 理由 |
|:---|:---|:---|
| Agent 模式 | 可选，默认工具模式 | 确定性优先 |
| 委托深度 | 单层委托，禁止转借 | 授权链条清晰 |
| Agent 拒绝权 | 有限拒绝（安全策略） | 工具本质+程序化边界 |
| 首次建立联系 | 人类必须介入一次 | 主权确认 |
| 感知层级默认 | L2 能力橱窗 | 促进有机连接 |
| 次级管理员 | 每节点 3 名，权限互斥 | 分散管理，防止权力集中 |
| Agent 监听 | 活跃/静默/离线/休眠四态 | 覆盖全场景可用性 |
| 离线队列 | 节点缓冲，恢复同步 | 异步友好，不丢消息 |
| 加密体系 | TLS + 契约 E2EE + Double Ratchet | 传输安全+内容保密+前向保密 |
| 密钥来源 | 契约派生，非临时协商 | 支持离线、Agent 自主、关系持久 |
| Agent 解密能力 | 仅解密给自己的指令 | 保护主人私密，Agent 是工具 |

---

> **Synch 的核心创新**：契约不是关系的描述，而是关系的可执行程序。Agent 成为社交原子，在人的主权边界内，编织能力流动的网络。

> **领主治理原则**：治理节点，但不侵犯主权；维护秩序，但不窥探隐私。

> **Agent 存在方式**：当主机在线，Agent 永远监听；当人类授权，Agent 自主执行；当契约超期，关系休眠待复活。

> **通讯安全原则**：契约即密钥，端到端加密；节点是信使，不是窃听者；Agent 是中介，不是解密者。
